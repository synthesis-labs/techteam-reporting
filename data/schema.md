# AI Adoption Data Model

Standardised model for monthly AI adoption reporting. Raw exports land in
`<YYYY-MM>/raw/`; `load.py` normalises them into `<YYYY-MM>/standardised/`.

Three adoption signals are tracked side-by-side:

- **License allocations** — who holds a seat in each AI tool (Claude, Cursor, ChatGPT, Copilot, Gemini)
- **Project AI usage** — what AI is being used on actual client work (from the PS Project Tracker)
- **Self-assessed maturity** — individual self-reported AI maturity (from the AI Maturity Self Assessment form)

All three are joined to the same `employees` master via email.

## Folder layout

```
data/
  schema.md                       # this file
  load.py                         # raw -> standardised loader
  aliases.csv                     # persistent email/handle -> employee_code lookup
  <YYYY-MM>/
    raw/                          # untouched tool exports, drop in as-is
      employees.csv               # HR employee master (or employee.xlsx — auto-detected)
      claude.csv
      cursor.csv
      chatgpt.csv                 # alternating-row format
      copilot.csv                 # alternating-row format with NBSP separator
      gemini.csv
      projects.csv                # PS Project Tracker export
      assessments.xlsx            # AI Maturity Self Assessment responses
    standardised/                 # generated by load.py
      employees.csv               # canonical employee master
      license_allocations.csv     # long-format: one row per license
      unmatched.csv               # rows that couldn't be resolved
      projects.csv                # one row per project, scores parsed
      project_ai_usage.csv        # long: one row per (project, ai_usage_tag)
      assessments.csv             # one row per assessment response, joined to employees
```

## Standardised tables

### `employees.csv`

One row per person. Source: `raw/employees.csv` after cleanup.

| Column           | Notes                                                      |
|------------------|------------------------------------------------------------|
| employee_code    | Primary key                                                |
| first_name       |                                                            |
| known_as         | Optional nickname column                                   |
| last_name        |                                                            |
| email            | Lowercased, trimmed. Bad values (`"0"`, blank, no `@`) preserved as-is and flagged in unmatched |
| job_title        | `OL - <text>` prefix stripped                              |
| department       | `<CODE> - <text>` prefix stripped                          |
| operating_level  | Raw OL code (e.g. `SENIOR`, `PQM`)                         |
| skill_level      | Mapped readable form (e.g. `Senior`, `Mid-level Manager`)  |

OL → skill_level map: `GRAD=Graduate, ASSOCIATE=Associate, SS=Semi-skilled, INTERMEDIATE=Intermediate, SENIOR=Senior, TECH=Tech Lead, PQM=Mid-level Manager, SM=Senior Manager, SPECIALIST=Specialist, PRINCIPAL=Principal, BU_HEAD=BU Head, DIR=Director`.

### `license_allocations.csv`

Long format. One row per (snapshot, tool, license seat). This is the source of
truth — the Excel workbook's wide matrix is a pivot of this table.

| Column          | Notes                                                              |
|-----------------|--------------------------------------------------------------------|
| snapshot_date   | ISO date of the export (YYYY-MM-DD)                                |
| tool            | `claude` \| `cursor` \| `chatgpt` \| `copilot` \| `gemini`         |
| source_email    | Email as it appears in the tool export, lowercased                 |
| source_name     | Name as it appears in the tool export                              |
| source_handle   | Copilot handle (NBSP-separated from name); blank for other tools   |
| employee_code   | Resolved employee_code, blank if unmatched                         |
| match_method    | `email` \| `alias` \| `name` \| `handle` \| `known_as` \| `unmatched` |
| raw_role        | Original Role field (e.g. `Owner`, `Member`, `Admin`)              |
| raw_seat_tier   | Original Seat Tier (Claude: `Standard`/`Premium`/`Unassigned`)     |
| raw_usage       | Stringified usage info (Cursor: `included/free/on_demand`)         |
| notes           | Free-text                                                          |

### `unmatched.csv`

Tool rows that didn't resolve to an employee_code. Drives Sheet 19 (Email
Verification) and Sheet 20 (Unmatched) in the workbook.

| Column                  | Notes                                              |
|-------------------------|----------------------------------------------------|
| snapshot_date           |                                                    |
| tool                    |                                                    |
| source_email            |                                                    |
| source_name             |                                                    |
| source_handle           |                                                    |
| reason                  | `bad_email_in_master` \| `no_match` \| `handle_unresolved` |
| suggested_employee_code | Auto-guess from name comparison, may be blank      |
| status                  | `confirm_fix` \| `ex_employee` \| `external` \| `shared` \| `investigate` \| empty |
| notes                   |                                                    |

### `projects.csv`

One row per project. Source: `raw/projects.csv`. Numeric scores parsed where possible;
text labels (e.g. `"5 - On schedule or ahead of schedule"`) preserved as-is — they
carry the rubric. The `AI Usage` JSON array is exploded into `project_ai_usage.csv`.

| Column                  | Notes                                                |
|-------------------------|------------------------------------------------------|
| snapshot_date           | YYYY-MM-01                                           |
| project_name            | Primary key within a snapshot                        |
| client                  |                                                      |
| bu                      | Business unit                                        |
| account_manager         | Display name as it appears in the tracker            |
| project_manager         | Display name                                         |
| tech_lead               | Display name from `Tech Lead / Core Member`          |
| lifecycle               | `Execution` \| `Handover` \| `On Hold` \| `Closed` \| `Working at Risk` |
| start_date              | YYYY-MM-DD when parseable, else blank                |
| end_date                | YYYY-MM-DD when parseable, else blank                |
| duration                | Free-text from tracker                               |
| overall_rating          | Float (comma→dot); blank for `#Name?`                |
| quality, team_confidence, budget_health, delivery_health, team_wellbeing, customer_satisfaction, scope_health | Raw labels |
| billing                 | `Fixed Billing` \| `Time & Materials`                |
| scope                   | `Variable Scope` \| `Fixed Deliverables`             |
| budget                  | Raw currency string from tracker                     |
| piia_review             | PII review status                                    |
| milestones              | Free-text                                            |
| documentation           | URL/path                                             |
| linked_projects         | Free-text                                            |
| high_care               | bool                                                 |
| budget_score, delivery_score, team_score, csat_score, scope_health_score | int, blank for `#Name?` |
| active                  | bool                                                 |
| modified                | YYYY-MM-DD                                           |
| modified_by             | Display name                                         |
| risk_issue_status       | Raw label                                            |

### `project_ai_usage.csv`

Long format. One row per (snapshot, project, ai_usage_tag). This is the
project-level adoption signal.

| Column         | Notes                                              |
|----------------|----------------------------------------------------|
| snapshot_date  |                                                    |
| project_name   |                                                    |
| ai_usage_tag   | One of: `No AI Usage`, `To be determined`, `Research Mode`, `AI for Development`, `AI in CI/CD`, `Complete Feature Built in AI` (additional tags pass through as-is) |

Projects with empty/missing AI Usage produce no rows here.

### `assessments.csv`

One row per AI Maturity Self Assessment response. Joined to `employees` by
respondent email where possible. Free-form question columns from the form are
passed through with normalised snake_case headers.

| Column           | Notes                                                       |
|------------------|-------------------------------------------------------------|
| snapshot_date    |                                                             |
| submitted_at     | ISO timestamp from form                                     |
| submitted_email  | Lowercased+trimmed                                          |
| employee_code    | Resolved via email match; blank if unmatched                |
| match_method     | `email` \| `unmatched`                                      |
| self_level       | Self-assigned maturity level (Level 0–4), if asked          |
| (other columns)  | Pass-through, snake_case'd from the form's question labels  |

### `aliases.csv` (persistent, not per-month)

Manual lookup for cases the auto-matcher can't resolve. Edit by hand; the
loader reads it on every run.

| Column        | Notes                                                  |
|---------------|--------------------------------------------------------|
| source_value  | The email or handle as seen in the tool export         |
| source_type   | `email` \| `handle`                                    |
| employee_code | Target employee in the master                          |
| notes         | Why this alias exists (e.g. "Copilot handle for Tom")  |

## Matching rules (applied in order)

1. **Direct email** — lowercased+trimmed equality against `employees.email`.
2. **Alias lookup** — `aliases.csv` by `(source_value, source_type)`.
3. **Normalised full name** — `first_name + last_name` vs source name.
4. **Known-as + last name** — same as above but using `known_as`.
5. **Copilot handle** — only via alias; unresolved handles go to unmatched.

Anything not resolved by step 5 lands in `unmatched.csv` with `reason=no_match`.

## Source quirks

- **Claude** (`Name, Email, Role, Status, Seat Tier`) — direct email.
- **Cursor** (`Name, Email, Role, Included Usage, Free Usage, On-Demand Usage`) —
  direct email. `Free Usage` may contain `"$20+"` (string) mixed with numbers.
- **ChatGPT** — alternating rows. Odd: `Name / Role / Tool / Date`. Even: email.
  Pair them on load.
- **GitHub Copilot** — alternating rows. Non-`@` rows are `Name<NBSP>Handle`
  (separator is U+00A0). Resolve handles via `aliases.csv`.
- **Gemini** — direct email from `Email Address [Required]`.

## Adding a new tool

1. Add a `load_<tool>()` function in `load.py` that yields the common dict shape.
2. Register it in the `LOADERS` map.
3. Add a row in the source quirks section above.
4. No schema change required — long format absorbs new tools.
