# TechTeam Reporting

Monthly AI adoption reporting for Synthesis Software Technologies. Standardises
license, project and assessment exports into a queryable data model and renders
a dashboard that publishes to GitHub Pages.

## Repos

This is a **two-repo** setup:

- **`synthesis-labs/techteam-reporting`** (this repo, public) — code, workflow,
  schema, and the aggregated history that drives the public dashboard.
- **`synthesis-labs/techteam-reporting-data`** (private) — all source data with PII:
  raw monthly exports, standardised CSVs, and `aliases.csv`.

CI fetches the private repo via deploy key (see [Setup](#setup-private-data-repo--ci-key)),
stages it into `data/`, then runs the same build as a local dev would.

## Layout

```
# In this public repo
data/
  schema.md                       Data model documentation
  load.py                         raw → standardised loader (per month)
  aggregate.py                    standardised → history (cross-month, PII-free)
  sync_private.sh                 pulls private data into data/ for local dev
  history/                        generated, PII-free aggregates (committed)
  .gitignore                      blocks <YYYY-MM>/ and aliases.csv from this repo

dashboard/
  build.py                        regenerates snapshot.md (current month)
  build_trends.py                 regenerates trends.md (long-term view)
  snapshot.md                     current-month Quarto source (committed for CI)
  trends.md                       long-term Quarto source (committed for CI)
  _quarto.yml                     renders both → ../dist/dashboard
  style.css                       Synthesis branding

# In the private techteam-reporting-data repo
<YYYY-MM>/
  raw/                            untouched monthly exports (drop in as-is)
  standardised/                   generated by load.py
aliases.csv                       persistent email/handle → employee_code lookup
```

Published URLs:
- **https://synthesis-labs.github.io/techteam-reporting/dashboard/dashboard.html** — long-term trends (rendered from `trends.md`)
- **https://synthesis-labs.github.io/techteam-reporting/dashboard/snapshot.html** — current-month snapshot

See [data/schema.md](data/schema.md) for the data model.

## Monthly workflow

All raw + standardised data lives in the **private** `techteam-reporting-data`
repo. The monthly steps happen there; this public repo only receives the
regenerated aggregates and dashboard sources.

1. Sync the private repo into `data/` (local working copy only — `data/<YYYY-MM>/`
   is gitignored in this repo):
   ```bash
   ./data/sync_private.sh
   ```

2. Drop raw exports into `_data_private/<YYYY-MM>/raw/`:
   - `employees.csv` (or `employee.xlsx`) — HR master
   - `claude.csv`, `cursor.csv`, `chatgpt.csv`, `copilot.csv`, `gemini.csv` — tool exports
   - `projects.csv` — PS Project Tracker
   - `assessments.xlsx` — AI Maturity Self Assessment responses

3. Standardise (writes back into `_data_private/<YYYY-MM>/standardised/`):
   ```bash
   ./data/sync_private.sh                   # stage into data/
   python data/load.py --month <YYYY-MM>    # standardise
   # copy generated standardised/ back to _data_private/<YYYY-MM>/standardised/
   rsync -a data/<YYYY-MM>/standardised/ _data_private/<YYYY-MM>/standardised/
   ```

4. Resolve unmatched rows by adding entries to `_data_private/aliases.csv`,
   re-stage with `./data/sync_private.sh`, and re-run the loader.

5. Commit raw + standardised + aliases.csv inside the **private repo** and push.

6. Regenerate the public artefacts in this repo:
   ```bash
   python data/aggregate.py                       # data/history/*.csv (PII-free)
   python dashboard/build.py --month <YYYY-MM>    # snapshot.md
   python dashboard/build_trends.py               # trends.md
   ```

7. Commit and push the public repo. CI fetches the private data, re-aggregates,
   re-renders, and publishes.

### Backfilling history

Drop older months' raw exports into `_data_private/<YYYY-MM>/raw/`, run
`python data/load.py --month <YYYY-MM>` against the staged copy, copy the
standardised output back into the private repo, push the private repo, then
re-run `python data/aggregate.py` and commit `data/history/` in this repo.

## Publishing

A push to `main` that touches `data/**` or `dashboard/**` triggers
[.github/workflows/build-dashboard.yml](.github/workflows/build-dashboard.yml).
The workflow:

1. Checks out this (public) repo
2. Checks out the **private** `techteam-reporting-data` repo using the
   `DATA_REPO_KEY` deploy key, then stages it into `data/`
3. Detects the latest snapshot
4. Aggregates history via `data/aggregate.py`
5. Regenerates `snapshot.md` via `dashboard/build.py`
6. Regenerates `trends.md` via `dashboard/build_trends.py`
7. Renders both with Quarto → `dist/dashboard/`
8. Pushes to the `gh-pages` branch at `/dashboard/` with `keep_files: true`

Trends becomes the main page (`dashboard.html`); the per-month snapshot lives
at `snapshot.html`. Both pages cross-link in their header.

### Setup: private data repo + CI key

One-time setup, only redo if you rotate the key:

1. **Create** a private repo `synthesis-labs/techteam-reporting-data`. Empty
   is fine.
2. **Populate** it with the layout under [Layout](#layout) — `<YYYY-MM>/raw/`,
   `<YYYY-MM>/standardised/`, and `aliases.csv` at the repo root.
3. **Generate a deploy key** for the data repo:
   ```bash
   ssh-keygen -t ed25519 -C "techteam-reporting CI" -f /tmp/data_repo_key -N ""
   ```
4. In **`techteam-reporting-data` → Settings → Deploy keys → Add deploy key**:
   paste the contents of `/tmp/data_repo_key.pub`. **Allow write access only
   if you also want CI to write back** (not needed for current workflow — keep
   read-only).
5. In **`techteam-reporting` → Settings → Secrets and variables → Actions**:
   add a new repository secret named **`DATA_REPO_KEY`** with the contents of
   `/tmp/data_repo_key` (the private key).
6. Delete the local key files: `rm /tmp/data_repo_key /tmp/data_repo_key.pub`.

### Trigger CI from a data push (optional)

To rebuild the dashboard automatically when the **private** data repo is
updated (instead of waiting for a push to this public repo), add this to the
data repo's `.github/workflows/notify.yml`:

```yaml
on:
  push:
    branches: [main]
jobs:
  notify:
    runs-on: ubuntu-latest
    steps:
      - uses: peter-evans/repository-dispatch@v3
        with:
          token: ${{ secrets.PUBLIC_REPO_PAT }}
          repository: synthesis-labs/techteam-reporting
          event-type: data-updated
```

You'll need to create a fine-grained PAT with Actions write access on the public
repo and store it as `PUBLIC_REPO_PAT` in the private repo's secrets.

### One-time GitHub Pages setup

After the first workflow run creates the `gh-pages` branch:

1. **Settings → Pages → Source** → "Deploy from a branch" → `gh-pages` / `(root)` → Save.
2. Wait ~30s; the URL above goes live.

## Local preview

Render both pages locally (requires [Quarto](https://quarto.org) and
`pip install plotly`):

```bash
./data/sync_private.sh                         # pull data from private repo
python data/aggregate.py
python dashboard/build.py --month <YYYY-MM>
python dashboard/build_trends.py
cd dashboard && quarto render
open ../dist/dashboard/dashboard.html          # trends
open ../dist/dashboard/snapshot.html           # snapshot
```
